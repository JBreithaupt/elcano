The Elcano Project aims to make self-drive real for students and hobbyists by developing low-cost hardware and software kits to convert any vehicle to self-drive. Elcano uses recumbent tricycles as a test bed. Finished Elcano tricycles can carry a person and cost under 5000 USD. Elcano kits could also be used in full-sized cars or in toy remote-controlled cars. The ElcanoProject.Org homepage outlines the project's greater purpose and goals.

System Hardware[edit]

The Elcano team has two self-driving tricycles for development and demonstration. The Elcano team purchased consumer recumbent tadpole tricycles and customized each trike to drive itself using technology outlined in this guide.

Trike Architecture Overview[edit]

Low-level control directly controls how the vehicle moves. These are the motors and computers that control steering, braking, and throttle. Each Elcano tricycle uses different low-level configurations.

High-level control makes decisions about where the vehicle should go. High-level control is not vehicle specific: each Elcano tricycle shares a high-level control configuration.

Actuators[edit]

Go to page A1: Traction Motor

A1 – Traction Motor The A1 motor turns the drive wheel. C1 controls the speed of A1. Depending on motor design, A1 may generate a motor speed feedback signal. Depending on the hub motor controller design, C1 may directly receive this feedback from A1 to determine how fast the wheel turns. If A1 has motor speed feedback but C1 is not connected to the feedback signal, C2 could receive the motor speed feedback to determine wheel speed and adjust control signals sent to C1.

Go to page A2: Braking Motor

A2 – Brake motor A2 is attached to the brakes and receives braking input from C2. Elcano currently uses a linear servo to control the position of disk brake calipers on the front wheels. C2 controls position of A2 using a PWM signal.

Go to page A3: Steering Motor

A3 – Steering motor A3 is attached to the steering and receives steering input from C2. Elcano currently uses a linear servo to control the position of the front wheels. C2 controls position of A3 using a PWM signal.

Processors[edit]

a. C1 – Hub motor controller
i. C1 documentation
ii. related code links
b. C2 – Dual control: low level vehicle control, either from the driver or the AI.
i. Documentation

C2 is the low-level controller that sends control signals to the hub motor, brake servo, and steering servo. It is a PID controller, but may also impose limits on control values for the motor and servos as a safety measure to protect against incorrect PID settings.

It receives desired speed and heading from either of two sources, an RC controller operated by a person, or the C3 pilot module. These are mutually exclusive.

RC commands are received directly as interrupts on a bank of pins. C3 commands are received over a serial line using the Elcano Serial protocol. Heading and speed commands do not need to be passed through to other modules, but because the Elcano Serial protocol uses a unidirectional ring structure, C2 may need to pass through *other* commands that come from C3 but are intended for modules past C2 on the ring.

Elcano #1 Servo range is 50 mm for brake, 100 mm for steering.

Elcano servo has a hardware controller that moves to a particular position based on an input PWM signal from Arduino. The Arduino PWM signal is a square wave at a base frequency of 490 Hz or 2.04 ms. PWM changes the duty cycle to encode 0 is always off; 255 always on. One step is 7.92 us.

Elcano servo is fully retracted on a pulse width of 2 ms; fully extended at 1 ms and centered at 1.5 ms. There is a deadband of 8 us. At 12v, servo operating speed is 56mm/s with no load or 35 mm/s at maximum load. 


ii. code
c. C3 – Pilot: Finds path around obstacles and feeds next path settings to C2.
i. Documentation

The Pilot program reads a serial line that specifies the desired path and speed of the vehicle. It computes the analog signals to control the throttle, brakes and steering and sends these to C2. Input will be received and sent using Elcano_Serial. Inputs will be received fromC5(sensors) and C4(planner). Output is sent to C2 (Low Level). In USARSIM simulation, these could be written on the serial line as wheel spin speeds and steering angles needed to follow the planned path. This would take the form of a DRIVE command to the C1 module over a serial line. The format of the DRIVE command is documented in the C1 code.

Wiring / connectors[edit]

a. Power wiring
b. Servo wiring

Output from hardware servo controller to either servo has five wires, with observed behavior of: White: 0V Yellow: 5V Blue: 0-5V depending on position of servo. Black: 12V while servo extends; 0V at rest or retracting. Red: 12V while retracting; 0V at rest or extending. The reading on the Blue line has hysteresis when Elcano sends a PWM signal; there appear to be different (PWM, position) pairs when retracting or extending. Motor speed is probably controlled by the current on the red or black line.
